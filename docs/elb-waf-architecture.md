## 架构说明

### 流量路径

WAF 独享模式强制要求此架构，ELB 专为 WAF 实例服务，不是为 BMS 做负载均衡。BMS 同时保留独立公网 EIP，用于直连对比测试：

```
                        ┌─────────────────────────────────┐
                        │         路径 A（生产路径）         │
客户端                   │                                  │
  │ HTTPS 443            │  ELB（独享型，新申请 EIP-B）       │
  │ maas.ovaijisuan.com  │    │ HTTP 内网转发                │
  ▼                      │    ▼                             │
[DNS A → EIP-B] ────────▶  WAF 独享实例 × 2（VPC 内）       │
                        │    │ HTTP 回源 → BMS:3000         │
                        └────┼─────────────────────────────┘
                             │
                             ▼
                        BMS:3000（new-api，docker-compose）
                             │
                        ┌────┼─────────────────────────────┐
                        │    │ 路径 B（直连路径，仅测试用）  │
                        │    ▼                             │
                        │  BMS 公网 EIP-A 111.172.214.58   │
                        │  安全组：3000 端口仅对测试 IP 开放 │
                        └─────────────────────────────────┘
                             │
                        ├──▶ RDS PostgreSQL（主备，10.0.0.73 / 10.0.0.239）
                        └──▶ Redis（BMS 容器内，与 new-api 同机）
```

### 各层职责

| 层 | 组件 | 职责 |
|---|---|---|
| 接入层 | ELB | SSL 终止、HTTP→HTTPS 重定向、将流量分发给 WAF 实例 |
| 安全层 | WAF 独享 × 2 | Web 攻击过滤、CC 防护、IP 黑白名单、回源到 BMS |
| 计算层 | BMS（bms-newapi） | 运行 new-api 容器，处理 AI 请求 |
| 数据层 | RDS | 主备高可用，由云平台管理故障切换 |
| 缓存层 | Redis（BMS 容器） | 单容器，与 new-api 同机运行，BMS 重启时整体不可用，风险可控 |

## 双路径对比方案

BMS 保留自己的独立公网 EIP（EIP-A），ELB 申请新的独立 EIP（EIP-B），两条路径同时运行，互不干扰。

| | 路径 A（直连） | 路径 B（WAF 路径） |
|---|---|---|
| 访问方式 | `http://BMS公网IP:3000` | `https://maas.ovaijisuan.com` |
| 链路 | 客户端 → BMS | 客户端 → ELB → WAF实例 → BMS |
| SSL | 无（HTTP） | 有（ELB 终止 TLS） |
| WAF 防护 | 无 | 有 |
| 用途 | 功能对比测试、问题排查 | 生产流量 |

### BMS 安全组策略（双路径并行期间）

基于实际安全组 `sg-newapi` 的变更操作：

**需要新增的规则：**

| 协议 | 来源 | 端口 | 说明 |
|---|---|---|---|
| TCP | 10.0.0.0/24（WAF 实例与 BMS 同子网） | 3000 | WAF 回源访问 new-api |

**需要删除的规则：**

| 协议 | 来源 | 端口 | 原因 |
|---|---|---|---|
| TCP | 0.0.0.0/0 | 80 | nginx 已停止，ELB 接管 80 端口 |
| TCP | 0.0.0.0/0 | 443 | nginx 已停止，ELB 接管 443 端口 |
| TCP | 0.0.0.0/0 | **32409** | **高危：全网开放的 NodePort，需立即调查并关闭** |

**需要保留的规则：**

| 协议 | 来源 | 端口 | 说明 |
|---|---|---|---|
| TCP | 111.4.10.245/32 | 3000 | 直连对比测试（对比完成后删除） |
| TCP | 171.83.85.95/32, 111.4.10.245/32 | 22 | SSH 运维 |
| ALL | 172.16.0.0/16, 173.2.0.0/16 | ALL | 对等连接内网互通（保留） |
| ALL | sg-newapi 自身 | ALL | 安全组内互通（保留） |

> **关于端口 32409**：该端口在 Kubernetes NodePort 范围（30000-32767）内，当前对 0.0.0.0/0 全网开放，属于高危配置。在调查清楚用途之前不要关闭，确认无业务影响后立即删除该规则。

> 直连路径绕过了 WAF，因此 3000 端口务必**只对测试 IP 开放**，不能对 0.0.0.0/0 开放，否则等同于裸奔。

### 对比测试重点关注项

| 测试项 | 直连路径预期 | WAF 路径关注点 |
|---|---|---|
| Streaming 响应 | 正常流式输出 | 观察是否被截断，是否缓冲后一次性返回 |
| 首 token 延迟 | 基准值 | WAF 路径额外延迟应 < 5ms（内网转发） |
| 长时间请求（> 2 分钟） | 正常完成 | 验证超时配置是否生效 |
| WebSocket 连接 | 正常 | WAF/ELB 均默认支持 WS，验证无需额外配置 |
| 文件上传 | 正常 | WAF 路径注意请求体大小限制 |
| 管理台登录 / OAuth | 正常 | 验证 Cookie、重定向、真实 IP 透传是否正常 |
| 频率限制 | 触发限流 | WAF CC 防护不能误拦截正常 API key 高频调用 |

### 对比完成后收尾

确认 WAF 路径一切正常后：

1. 将 BMS 安全组 3000 端口的测试 IP 来源规则**删除**（只保留 WAF 子网回源）
2. BMS 的公网 EIP-A 可以保留（仅用于 SSH 运维），不再对外提供应用服务

## 第一阶段：ELB 配置

ELB 控制台 → 购买弹性负载均衡 → 选择**独享型**

| 参数 | 值 |
|---|---|

| 所在 VPC | vpc-newapi |

暂不绑定 EIP，先完成监听器配置。

### 2. 创建后端服务器组（WAF 实例用）

"后端服务器组" → 创建后端服务器组

| 参数 | 值 | 说明 |
|---|---|---|
| 后端协议 | HTTP | WAF 实例对外暴露 HTTP 端口 |
| 分配策略 | 加权轮询 | 两个 WAF 实例均等分流 |
| 会话保持 | **关闭** | AI API 无状态，开启反而影响 WAF 实例升级 |
| 健康检查协议 | HTTP | |
| 健康检查路径 | `/` | WAF 实例根路径可达即可 |
| 健康检查间隔 | 10 秒 | |
| 健康检查超时 | 5 秒 | |
| 最大重试次数 | 3 次 | 连续 3 次失败才摘除 WAF 实例 |

添加后端服务器：将两个 WAF 独享实例的**内网 IP** 加入，端口填 WAF 实例 HTTP 监听端口（通常为 **80**）。

---

### 3. 创建 HTTPS 监听器（443 端口）

"监听器" → 添加监听器

**基本配置：**

| 参数 | 值 |
|---|---|
| 前端协议 | HTTPS |
| 前端端口 | 443 |
| SSL 证书 | 上传 `maas.ovaijisuan.com` 的域名证书（PEM 格式） |
| TLS 安全策略 | TLS-1-2（兼容性与安全性平衡） |
| HTTP/2 | **开启**（提升多路复用性能，对 streaming 有益） |

**高级配置（关键超时参数）：**

| 参数 | 推荐值 | 范围 | 原因 |
|---|---|---|---|
| 空闲超时时间 | **3600 秒** | 0~4000s | AI streaming 推理期间可能较长时间无新数据包，但连接不能断 |
| 请求超时时间 | **300 秒** | 1~300s | 客户端上传请求体的最大时间，设为上限值 |
| 响应超时时间 | **300 秒** | 1~300s | ELB 等待 WAF 实例首个响应头的超时，设为上限值 |

> ELB 文档说明：修改后端服务器权重为 0 后，已有的长连接在响应超时时间到期前仍继续转发，不会立即断连。HTTP/HTTPS 监听器的长连接会在响应超时时间后才断开。

后端：选择刚才创建的 WAF 服务器组，**后端协议 HTTP**，**后端端口 80**。

---

### 4. 创建 HTTP 监听器（80 端口，重定向）

"监听器" → 添加监听器

| 参数 | 值 |
|---|---|
| 前端协议 | HTTP |
| 前端端口 | 80 |
| 默认动作 | **重定向到 HTTPS**，目标端口 443，状态码 301 |

不配置后端服务器组，所有 HTTP 请求直接重定向，不转发到 WAF 实例。

---

### 5. 为 ELB 申请并绑定新 EIP（EIP-B）

在 ELB 基本信息页面绑定 EIP。

**重要**：申请一个**新的 EIP（EIP-B）**，不要动 BMS 原有的 EIP（EIP-A）。BMS 保留 EIP-A 用于直连对比测试和 SSH 运维。

- EIP-B 绑定到 ELB，作为 `maas.ovaijisuan.com` 的生产入口
- DNS 将 `maas.ovaijisuan.com` 的 A 记录改为指向 EIP-B

### 3. 更新 DNS 指向 ELB（步骤三）

ELB 已绑定新的 EIP-B，BMS 保留原有 EIP-A，只需更新 DNS：

1. DNS 控制台 → 将 `maas.ovaijisuan.com` 的 A 记录改为 **EIP-B**（ELB 的 EIP）
2. TTL 建议提前改为 60 秒，等 DNS 生效后再改回 300 秒
3. 生效后通过域名访问走 WAF 路径，通过 `BMS公网IP:3000` 访问走直连路径

切流完成后两条路径并行，可同时对比测试。

---

### 4. 配置超时（关键步骤）

网站设置 → 点击 `maas.ovaijisuan.com` → 找到"超时配置"行

点击开关**开启**（开启后**不可关闭**，确认再操作）：

| 参数 | 推荐值 | 原因 |
|---|---|---|
| 连接超时 | **30 秒** | WAF 与 BMS 建立 TCP 连接的超时，默认值足够 |
| **读超时** | **300 秒** | WAF 从 BMS 读取响应的超时，AI streaming 响应可能持续数分钟，**必须调大** |
| 写超时 | **60 秒** | WAF 向 BMS 发送请求体的超时 |

> WAF 文档说明：WAF 到源站的连接超时默认仅 30 秒。浏览器到 WAF 的超时为 120 秒（固定，不可修改），但该超时为连接超时而非空闲超时，streaming 期间数据持续流动不会触发。

### 5. 配置防护策略

防护策略 → 选择关联 `maas.ovaijisuan.com` 的策略

| 规则 | 建议设置 | 说明 |
|---|---|---|
| Web 基础防护 | 开启，模式先设**仅记录** | 观察 1-2 天确认无误报后再切换为拦截模式 |

| 爬虫防护 | **谨慎开启** | AI API SDK 的请求头与浏览器不同，容易被误判为爬虫 |
| IP 黑名单 | 按需添加已知恶意 IP 段 | |

## 第三阶段：BMS 侧配置

### 0. 停止 nginx

当前 BMS 上的 nginx 承担 SSL 终止和反代职责（443 → 127.0.0.1:3000）。引入 ELB+WAF 后，ELB 接管 SSL 终止，WAF 直接回源到 BMS:3000，nginx 不再在流量链路中，必须停止以释放端口并避免干扰。

```bash
# 停止并禁用 nginx，防止重启后自动启动
sudo systemctl stop nginx
sudo systemctl disable nginx

# 确认 443 端口已释放
sudo ss -tlnp | grep 443
# 期望：无任何输出（端口已空闲）
```

> **操作时机**：在 DNS 切换至 EIP-B（ELB）之后再停 nginx。DNS 切换前 nginx 仍需服务旧流量。

---

### 1. docker-compose.yml 端口绑定

安全组已收紧，将端口绑定改为全网卡监听（允许 WAF 实例访问）：

```yaml
# 修改前（仅本机访问）
ports:
  - "127.0.0.1:3000:3000"

# 修改后（通过安全组控制来源，不再依赖 IP 绑定）
ports:
  - "3000:3000"
```

### 2. Gin TrustedProxies 配置

WAF 回源后，BMS 收到的请求源 IP 是 WAF 实例的内网 IP，需要配置 Gin 信任 WAF 子网，否则 `c.ClientIP()` 返回的是 WAF IP 而非用户真实 IP，影响限流和审计日志。

找到 Gin engine 初始化位置，确认已设置：

```go
// WAF 独享实例与 BMS 同在 10.0.0.0/24 子网
// Gin 信任该子网，才会从 X-Forwarded-For 中取用户真实 IP
// 否则 c.ClientIP() 返回 WAF 内网 IP，影响限流和审计日志
r.SetTrustedProxies([]string{"10.0.0.0/24"})
```

> 如果后续 WAF 实例迁移到独立子网，需同步修改此处 CIDR。

## 在线升级操作

### 背景

ELB 文档说明：将后端服务器权重改为 0 后，**已有长连接（含 streaming 请求）在响应超时时间到期前继续转发，新请求不再分发**，实现优雅摘流。

### 单台 BMS 的最优升级流程

当前只有一台 BMS，无法实现真正零停机，但可以将中断时间压缩到 5~10 秒：

```bash
# 1. 通知 Watchtower 拉取最新镜像（后台拉取，不立即重启）
docker pull <new-api镜像>

# 2. 优雅停机：给 new-api 30 秒处理完当前请求再停止
docker stop --time=30 new-api

# 3. 启动新版本
docker compose --env-file .env.prod up -d

# 4. 验证新版本正常
curl http://127.0.0.1:3000/api/status
```

**建议**：升级窗口选在业务低谷（如凌晨 2:00~6:00，与 RDS 维护窗口一致），并提前在公告页通知用户。

### 后续双节点滚动升级（扩容后）

扩容到 2 台 BMS 后可实现零停机滚动升级：

```
1. WAF 控制台：将 BMS-A 的源站权重改为 0（流量全走 BMS-B）
   等待 BMS-A 当前 streaming 请求自然完成（最长等响应超时 300 秒）
2. 升级 BMS-A：docker stop --time=30 && docker compose up -d
3. 验证 BMS-A 正常后，恢复其权重
4. 对 BMS-B 重复上述步骤
```

## 多实例兼容性分析

扩展到多节点前必须处理以下问题。

### ❌ 必须处理：BATCH_UPDATE 导致额度双花

`BATCH_UPDATE_ENABLED=true` 时，每个节点将 quota 扣减先累积在**进程内存**中，每 5 秒批量写一次数据库。两节点的内存互相不可见，导致以下场景：

```
Node A 内存：user_1 待扣 -500 token（未刷库）
Node B 内存：user_1 待扣 -500 token（未刷库）
Redis 缓存：user_1 quota = 1000（两节点读到相同值）

→ 两节点都认为用户有余额，都放行请求
→ 两节点各自刷库：DB quota = 1000 - 500 - 500 = 0
→ 用户实际被消费 1000 token，但两节点都只看到自己扣的 500
```

**解决方案：** 多节点时关闭 batch update：

```env
BATCH_UPDATE_ENABLED=false
```

性能影响：每次请求实时写 DB，写入量增加。RDS 规格（32 核 64GB）完全可以承受。

## 注意事项

**证书管理**

- ELB 侧：上传证书用于 SSL 终止，向客户端提供 HTTPS
- WAF 侧：上传同一张证书用于解密 HTTPS 流量进行内容检测
- 同一张 `maas.ovaijisuan.com` 证书，两处各上传一份
- 证书到期前在两处同步更新，WAF 支持配置证书到期告警

**RDS 主备跨可用区（待处理）**

当前 RDS 主备均在可用区 1，建议申请将备库迁移到可用区 2 或 3，实现真正的跨可用区容灾。主库故障时备库才能在不同物理机房接管。

**WAF 回源 IP 白名单**

WAF 独享实例的回源 IP 是其所在子网 IP，不是固定地址。如果后续 WAF 实例扩容或更换，需重新确认子网 CIDR 并更新 BMS 安全组规则。
