# ELB + WAF 高可用接入层设计

在现有云托管 PostgreSQL + Redis 主备方案基础上，通过 ELB（弹性负载均衡）和 WAF（Web 应用防火墙）实现接入层高可用与安全防护。

## 架构设计

### 当前状态

```
用户请求
    │
    ▼
[DNS] A 记录 → 服务器公网 IP
    │
    ▼
[ECS 服务器]
├── new-api 容器 (127.0.0.1:3000)
└── watchtower 容器
```

### 目标状态

```
用户请求
    │
    ▼
[DNS] CNAME → WAF 接入域名
    │
    ▼ 过滤 SQL 注入 / XSS / CC 攻击 / IP 黑名单
[WAF（云模式）]
    │
    ▼ 转发洁净流量到 ELB 公网 EIP
[ELB 应用型负载均衡]
    │  ├── HTTPS 443 监听（SSL 证书在此终止）
    │  └── HTTP 80 → 重定向 443
    │
    ▼ 内网 HTTP 转发（健康检查 /api/status）
[ECS 服务器 × N]
    │  安全组仅允许 ELB 后端子网 → 3000 端口
    ▼
[new-api 容器 :3000]
    │
    ├──▶ [云 PostgreSQL RDS 主备]
    └──▶ [云 Redis 主备]
```

### 各层职责

| 层 | 组件 | 职责 |
|---|---|---|
| 接入层 | WAF | Web 攻击过滤、CC 防护、IP 黑白名单 |
| 负载层 | ELB | SSL 终止、流量分发、健康检查、故障摘除 |
| 计算层 | ECS × N | 无状态应用容器，可随时横向扩展 |
| 数据层 | RDS + Redis | 主备高可用，由云平台管理故障切换 |

## WAF 配置

### 云模式 vs 独享模式

| | 云模式 | 独享模式 |
|---|---|---|
| 部署方式 | DNS CNAME 指向 WAF | WAF 实例在自己 VPC 内 |
| 配置复杂度 | 低 | 中 |
| 延迟 | 略高（多一跳） | 低（内网转发） |
| 适用场景 | 当前规模，快速上线 | 高流量或有合规要求 |

**建议先用云模式**，后续流量增长或有安全合规需求时再迁移到独享模式。

### 云模式配置步骤

1. WAF 控制台 → 网站接入 → 添加防护网站
2. 填写域名（如 `api.yourdomain.com`）
3. 源站地址填 **ELB 的公网 EIP**（不是服务器 IP）
4. 协议选 HTTPS，端口 443
5. WAF 会生成一个接入 CNAME，记录下来（DNS 变更时用）

### 建议开启的防护规则

| 规则类型 | 说明 |
|---|---|
| Web 基础防护 | 开启，防 SQL 注入、XSS、命令注入等 |
| CC 攻击防护 | 开启，针对 AI API 的高频调用场景尤其重要 |
| IP 黑名单 | 按需添加已知恶意 IP 段 |
| 精准访问控制 | 可针对 `/api/` 路径单独设置频率限制 |
| 地理位置封禁 | 按业务需要，屏蔽非服务区域的请求 |

## ELB 配置

### 创建负载均衡器

- 类型：**应用型（共享型或独享型均可）**
- 网络：公网，分配 EIP
- 所在子网：与 ECS 服务器同 VPC

### 后端服务器组

- 协议：HTTP
- 端口：3000
- 负载算法：加权轮询（默认即可）
- 健康检查：
  - 协议：HTTP
  - 路径：`/api/status`
  - 正常阈值：返回 HTTP 200 且响应体包含 `"success":true`
  - 检查间隔：10 秒
  - 超时：5 秒
  - 不健康阈值：连续 3 次失败后摘除

### 监听器配置

**HTTPS 监听器（443）**

- 协议：HTTPS
- SSL 证书：在 ELB 上传/绑定域名证书（SCM 证书管理服务）
- 后端协议：HTTP（内网无需加密）
- 开启 `X-Forwarded-For` 透传（将客户端真实 IP 传给后端）

**HTTP 监听器（80）**

- 协议：HTTP
- 动作：**重定向到 HTTPS**（不转发到后端）

### 安全组规则（ELB 侧）

| 方向 | 来源 | 端口 | 说明 |
|---|---|---|---|
| 入方向 | 0.0.0.0/0 | 443 | 接收 WAF 转发的 HTTPS 流量 |
| 入方向 | 0.0.0.0/0 | 80 | 接收 HTTP 流量后重定向 |

## 服务器侧改动

### docker-compose.yml 端口绑定

当前配置只允许本机访问，ELB 后端健康检查会失败：

```yaml
# 当前（需修改）
ports:
  - "127.0.0.1:3000:3000"

# 修改后（去掉 IP 限制，通过安全组控制来源）
ports:
  - "3000:3000"
```

### ECS 安全组规则（服务器侧）

修改后必须同步收紧安全组，防止 3000 端口暴露到公网：

| 方向 | 来源 | 端口 | 说明 |
|---|---|---|---|
| 入方向 | ELB 后端子网 CIDR | 3000 | 只允许 ELB 访问应用端口 |
| 入方向 | 运维 IP | 22 | SSH 管理 |
| 入方向 | 0.0.0.0/0 | — | **删除**对 3000 端口的公网放行规则 |

> ELB 后端子网 CIDR 在 ELB 控制台的「基本信息」中可以查到。

### 服务器不再需要公网 IP 对外暴露应用端口

用户只能通过 WAF → ELB 访问服务，服务器的公网 IP 仍可保留用于 SSH 运维，但 3000 端口对公网关闭。

## DNS 变更

| | 记录类型 | 记录值 |
|---|---|---|
| 变更前 | A | 服务器公网 IP |
| 变更后 | CNAME | WAF 提供的接入域名 |

变更步骤：

1. WAF 控制台获取接入 CNAME（格式类似 `xxxxxx.waf.huaweicloud.com`）
2. 在 DNS 服务商处将 `api.yourdomain.com` 的 A 记录改为 CNAME，值填 WAF 接入域名
3. TTL 建议提前改低（如 60 秒），变更生效后再改回

> WAF 云模式下，证书也需要在 WAF 侧上传一份（WAF 需要解密流量才能检测内容）。ELB 侧同样需要一份。同一张证书上传两处即可。

## 真实 IP 透传

流量经过 WAF → ELB 两层转发后，服务器收到的请求 IP 是 ELB 的内网 IP，而非客户端真实 IP。这会影响应用的限流、审计日志、IP 黑名单等功能。

### 透传链路

```
客户端 IP → WAF（追加 X-Forwarded-For） → ELB（追加/扩展 X-Forwarded-For） → new-api
```

ELB 开启 `X-Forwarded-For` 后，请求头格式：

```
X-Forwarded-For: <客户端真实 IP>, <WAF IP>, <ELB IP>
```

### Gin 框架配置

new-api 使用 Gin，需要配置信任代理，否则 `c.ClientIP()` 返回的是 ELB IP 而不是真实客户端 IP。

需要确认代码中是否已设置 `TrustedProxies`，找到 Gin engine 初始化的位置：

```go
// 信任 ELB 所在子网，Gin 才会从 X-Forwarded-For 中取真实 IP
r.SetTrustedProxies([]string{"ELB后端子网CIDR"})
// 例如：r.SetTrustedProxies([]string{"192.168.0.0/24"})
```

若设为 `nil` 则信任所有代理（不推荐生产使用），若不设置则 Gin 默认信任 `127.0.0.1`。

## 横向扩展

WAF + ELB 方案天然支持横向扩展，扩容时 **DNS 不动、WAF 不动**，只需：

1. 新建 ECS，安装 Docker
2. 将 `.env.prod` 和 `docker-compose.yml` 复制过去
3. `docker compose --env-file .env.prod up -d`（new-api 连同一套云 PG + Redis）
4. 在 ELB 后端服务器组中添加新 ECS，设置权重
5. ELB 健康检查通过后自动开始分流

因为数据层（PostgreSQL、Redis）是独立的云服务，多个 new-api 实例共享同一套数据，`SESSION_SECRET` 一致保证 session 互通，无需配置 sticky session。

## 多实例兼容性分析（2026-04-26）

扩展到多节点前必须了解以下问题，部分需要在上线前处理。

### ❌ 必须处理：BATCH_UPDATE 导致额度双花

**问题根因：**

`BATCH_UPDATE_ENABLED=true` 时，每个节点将 quota 扣减先累积在**进程内存**中，每 5 秒批量写一次数据库。两节点的内存互相不可见，导致以下场景：

```
Node A 内存：user_1 待扣 -500 token（未刷库）
Node B 内存：user_1 待扣 -500 token（未刷库）
Redis 缓存：user_1 quota = 1000（两节点读到相同值）

→ 两节点都认为用户有余额，都放行请求
→ 两节点各自刷库：DB quota = 1000 - 500 - 500 = 0
→ 用户实际被消费 1000 token，但两节点都只看到自己扣的 500
```

**解决方案：** 多节点时在 `.env.prod` 中关闭 batch update：

```env
BATCH_UPDATE_ENABLED=false
```

性能影响：每次请求实时写 DB，写入量增加。RDS 规格（32 核 64G）完全可以承受。

---

### ❌ 必须处理：ELB 超时截断 AI 流式响应

**问题根因：**

华为云 ELB 默认空闲连接超时为 **60 秒**，而 AI streaming（SSE）单次请求可能持续 2～5 分钟，超时后 ELB 强制断连，客户端收到截断响应。

**解决方案：** ELB 控制台 → 监听器 → 修改「空闲超时」为 **300 秒**（大于应用的 `STREAMING_TIMEOUT` 默认值 120 秒）。

---

### ⚠️ 需要注意：Watchtower 在线升级只更新当前节点

**问题根因：**

管理台「立即安装」请求打到某一节点，只触发该节点本地的 `http://watchtower:8080`，其他节点不受影响，导致各节点版本不一致。

**解决方案：** 每次升级时在所有节点手动触发：

```bash
# 每台服务器各执行一次
curl -H "Authorization: Bearer <WATCHTOWER_API_TOKEN>" http://localhost:8080/v1/update
```

或采用滚动升级：先将 Node A 从 ELB 摘除 → 升级 → 加回 → 再操作 Node B。

---

### ✅ 兼容，无需处理

| 项目 | 结论 | 说明 |
|---|---|---|
| **管理台配置** | 完全兼容 | 所有配置写入 PostgreSQL `options` 表，其他节点通过后台 goroutine 每 60 秒同步一次内存，变更最多延迟 60 秒生效 |
| **Session / 登录态** | 完全兼容 | `SESSION_SECRET` 固定，JWT 无状态，无需 sticky session |
| **Redis 缓存** | 完全兼容 | 所有节点共享同一套外部 Redis，数据一致 |
| **长连接** | 可接受 | 流式请求绑定到当前节点，节点故障时客户端需重试，属正常 HA 代价 |
| **日志** | 可接受 | 日志写各节点本地，设不同 `NODE_NAME` 可区分来源；有需要再接入集中日志服务 |

## 注意事项

**WAF 回源 IP**

WAF 云模式下，ELB 收到的请求来源 IP 是 WAF 的回源 IP 段，不是客户端 IP。需要在 ECS 安全组和 ELB 安全组上放行 WAF 的回源 IP 段（华为云 WAF 文档中有对应 IP 列表）。

**证书管理**

- WAF 侧：上传证书用于解密 HTTPS 流量进行内容检测
- ELB 侧：上传证书用于 SSL 终止，向客户端提供 HTTPS
- 同一域名证书，两处各上传一份
- 证书到期前在两处同步更新

**健康检查路径**

ELB 健康检查使用 `GET /api/status`，该接口不需要鉴权，当前代码已满足要求。健康检查请求不会计入业务流量统计。

**WAF 绕过风险**

配置完成后，务必将服务器安全组的 3000 端口对外关闭，防止攻击者直接访问服务器 IP 绕过 WAF 防护。验证方法：

```bash
# 直接访问服务器 IP 应该超时或拒绝
curl -v http://<服务器公网IP>:3000/api/status

# 通过域名访问应该正常
curl -v https://api.yourdomain.com/api/status
```
